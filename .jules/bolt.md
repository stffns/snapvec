## 2025-04-15 - [np.einsum is slow for batched residual LUTs]
**Learning:** `np.einsum` in `_ivfpq.py` for computing the batch residual LUT (`np.einsum("bjs,jks->bjk", q_split, self._codebooks)`) is surprisingly slow compared to explicit transposes and `np.matmul` (`@`). Even with `optimize=True`, `np.einsum` is slower, but without it (as was the case originally), it's extremely slow because it does not map well to BLAS under the hood in NumPy for this specific memory layout. By explicitly transposing to `(M, B, d_sub)` and `(M, d_sub, K)`, we can leverage batched matrix multiplication (`@`), which maps down to a highly optimized BLAS call and yields a roughly 25-30% reduction in `search_batch` end-to-end latency for `IVFPQSnapIndex`.
**Action:** When performing batched tensor contractions over multiple axes in performance-critical code paths with NumPy, benchmark `np.einsum` against manual reshape/transpose + `np.matmul` (`@`), as the latter is often much faster since it directly maps to highly optimized BLAS routines like `gemm` or `gemv`.
